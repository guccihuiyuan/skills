# Keyblade Pro Workflow

## 技能概述

这是一个**严格的工作流编排器**，用于协调多个技能完成AI代码编写任务。核心原则是：

1. **不要一上来就写代码**：必须先进行需求分析和计划制定
2. **严格顺序执行**：必须按照 brainstorming → writing-plans → executing-plans → keyblade-pro-code-rules 的顺序执行
3. **用户确认必需**：每个步骤执行前和执行后均需用户确认，**不跳过、不自动默认**

**核心特性：**
- 严格的流程控制：确保步骤顺序执行
- 强制用户确认：每个环节都需要用户明确确认
- 技能自动检测：在执行每个步骤前检测依赖技能是否存在
- 自动跳过缺失技能：若某个技能不存在，自动跳过该环节并继续后续流程

## 何时触发

**必须立即使用此技能的情况：**
- 用户要求"编写代码"、"生成代码"、"实现功能"
- 用户要求"修改代码"、"重构代码"、"优化代码"
- 用户要求"创建组件"、"开发页面"、"实现接口"
- 用户提出新的功能需求或 bug 修复需求

## 工作流流程

### 步骤 1：需求分析 (brainstorming)

**执行前检测 `brainstorming` 技能是否存在**

**执行内容：**
- 深入理解用户需求
- 分析功能边界和技术选型
- 探索可能的实现方案
- 识别潜在风险和依赖

**输出要求：**
- 明确的需求文档
- 技术方案概述
- 关键设计决策

**用户交互（必需）：**
- **执行前询问**："即将开始需求分析，是否确认执行？"
  - 用户选择：确认继续 / 取消终止
- **执行后确认**："需求分析已完成，以下是分析结果摘要：\n[摘要内容]\n\n请问您是否确认此分析结果？"
  - 用户选择：确认继续 / 重新分析 / 补充需求

**技能缺失处理：**
- 若 `brainstorming` 技能不存在，提示："未检测到 brainstorming 技能，将跳过需求分析环节"
- 自动进入下一步

### 步骤 2：计划制定 (writing-plans)

**需求分析完成后，执行前检测 `writing-plans` 技能是否存在**

**执行内容：**
- 创建详细的开发计划
- 分解任务到可执行的单元
- 定义任务优先级和依赖关系
- 预估时间和资源需求

**输出要求：**
- 完整的任务列表
- 任务优先级排序
- 时间估算

**用户交互（必需）：**
- **执行前询问**："即将制定开发计划，是否确认执行？"
  - 用户选择：确认继续 / 返回上一步 / 取消终止
- **执行后确认**："开发计划已制定完成，包含 [N] 个任务，预计耗时 [X] 小时。任务列表摘要：\n[摘要内容]\n\n请问您是否确认此计划？"
  - 用户选择：确认继续 / 调整计划 / 重新制定

**技能缺失处理：**
- 若 `writing-plans` 技能不存在，提示："未检测到 writing-plans 技能，将跳过计划制定环节"
- 自动进入下一步

### 步骤 3：计划执行 (executing-plans)

**计划制定完成后，执行前检测 `executing-plans` 技能是否存在**

**执行内容：**
- 按照计划逐步执行任务
- 跟踪任务进度和完成状态
- 管理任务间的依赖关系
- 协调多步骤任务的执行顺序

**输出要求：**
- 任务执行进度报告
- 已完成任务清单
- 执行过程中的问题记录

**用户交互（必需）：**
- **执行前询问**："即将开始执行开发计划，是否确认执行？"
  - 用户选择：确认继续 / 返回上一步 / 取消终止
- **执行中反馈**：定期向用户汇报任务执行进度
- **执行后确认**："计划执行已完成，共执行了 [N] 个任务。执行摘要：\n[摘要内容]\n\n请问您是否确认执行结果？"
  - 用户选择：确认继续 / 重新执行 / 调整计划后重试

**技能缺失处理：**
- 若 `executing-plans` 技能不存在，提示："未检测到 executing-plans 技能，将跳过计划执行环节"
- 自动进入下一步

### 步骤 4：代码编写 (keyblade-pro-code-rules)

**计划执行完成后，执行前检测 `keyblade-pro-code-rules` 技能是否存在**

**执行内容：**
- 按照项目规范编写代码
- 生成类型定义、枚举、请求函数等
- 创建组件、页面、接口等
- 遵循 Vue3 + TypeScript + Arco Design 规范

**输出要求：**
- 完整的代码文件
- 符合规范的代码结构

**用户交互（必需）：**
- **执行前询问**："即将开始代码编写，是否确认执行？"
  - 用户选择：确认继续 / 返回上一步 / 取消终止
- **执行中确认**：对于关键设计决策（如文件结构、组件拆分、API 设计），逐一询问用户确认
- **执行后确认**："代码编写已完成，共创建/修改了 [N] 个文件。变更摘要：\n[摘要内容]\n\n请问您是否确认代码实现？"
  - 用户选择：确认继续 / 修改实现 / 重新编写

**技能缺失处理：**
- 若 `keyblade-pro-code-rules` 技能不存在，提示："未检测到 keyblade-pro-code-rules 技能，将跳过代码编写环节"
- 自动进入下一步

### 步骤 5：代码审查与验证

**代码编写完成后执行**

**执行内容：**
- 代码规范检查
- 类型安全检查
- 运行 lint 和 typecheck

**用户交互（必需）：**
- **执行前询问**："即将进行代码审查与验证，是否确认执行？"
  - 用户选择：确认继续 / 返回上一步 / 取消终止
- **执行后反馈**："代码审查完成，发现 [N] 个问题：\n[问题摘要]\n\n是否需要我修复这些问题？"
  - 用户选择：修复问题 / 跳过修复 / 手动处理

### 步骤 6：最终确认与交付

**所有步骤完成后执行**

**执行内容：**
- 汇总所有工作成果
- 展示完整交付物

**用户交互（必需）：**
- **交付摘要**："本次开发工作已全部完成，以下是交付成果：\n\n1. 需求分析文档\n2. 开发计划\n3. 代码文件（共 [N] 个）\n4. 代码审查报告\n\n是否确认交付？"
  - 用户选择：确认交付 / 返回修改 / 结束会话

## 工作流约束

### 必须遵守的规则
1. **严格顺序执行**：必须按照 brainstorming → writing-plans → executing-plans → keyblade-pro-code-rules 的顺序执行
2. **不要一上来就写代码**：必须先进行需求分析和计划制定
3. **技能检测优先**：每个步骤执行前必须检测依赖技能是否存在
4. **用户确认必需**：每个步骤执行前和执行后均需用户确认，**不跳过、不自动默认**
5. **用户可控**：用户可以在任意步骤选择暂停、返回或终止工作流
6. **进度透明**：向用户清晰展示当前进度和下一步计划
7. **输出驱动**：每个步骤必须产生明确的输出成果

### 步骤间的依赖关系

```
                                                              │
                                                              ▼
                                                         最终交付
```

## 核心依赖技能

| 技能名称 | 用途 | 调用时机 | 检测与跳过 |
|---------|------|---------|-----------|
| `brainstorming` | 需求分析和方案设计 | 工作流第一步 | 不存在时跳过，进入计划制定 |
| `writing-plans` | 创建开发计划和任务列表 | 需求分析完成后 | 不存在时跳过，进入计划执行 |
| `executing-plans` | 执行开发计划并跟踪进度 | 计划制定完成后 | 不存在时跳过，进入代码编写 |
| `keyblade-pro-code-rules` | 代码编写和规范检查 | 计划执行完成后 | 不存在时跳过，进入代码审查 |

## Vue3 + TypeScript + Arco Design 项目特殊处理

### 表格页开发流程
1. 需求分析：明确表格功能需求
2. 计划制定：分解任务（类型定义、请求函数、组件开发）
3. 代码编写：使用 kb-pro-table、useTable、useForm

### 组件开发流程
1. 需求分析：明确组件功能和接口
2. 计划制定：分解任务（props定义、事件处理、样式开发）
3. 代码编写：使用 arco design + unocss

## 输出规范

### 每个步骤必须输出
- **brainstorming**：需求文档、技术方案
- **writing-plans**：任务列表、优先级排序
- **executing-plans**：执行进度报告、完成任务清单
- **keyblade-pro-code-rules**：代码文件、测试文件

### 最终交付
- 完整的代码文件
- 代码审查报告
- 功能验证结果

## 错误处理

### 需要停止并询问用户的情况
- 需求不明确或存在歧义
- 技术栈无法识别
- 规范冲突无法解决
- 需要用户确认的设计决策
- 执行过程中遇到异常情况
- 用户主动要求暂停或终止

### 用户交互处理
- **需求歧义处理**：当需求不明确时，列出具体的疑问点，逐一询问用户确认
- **技术栈识别**：若无法识别项目技术栈，询问用户是否需要手动指定
- **设计决策确认**：对于关键设计决策，提供多个选项供用户选择
- **异常处理**：遇到异常时，向用户展示错误信息并提供解决方案选项

### 用户可控操作

| 操作 | 说明 | 可用时机 |
|------|------|---------|
| **确认继续** | 确认当前步骤成果，进入下一步 | 每个步骤完成后 |
| **返回上一步** | 返回上一个已完成步骤重新执行 | 任意步骤 |
| **重新执行** | 从当前步骤重新开始 | 执行中 |
| **取消终止** | 结束工作流，保存已完成成果 | 任意步骤 |

## 最佳实践

1. **严格遵循流程**：必须按照 brainstorming → writing-plans → executing-plans → keyblade-pro-code-rules 的顺序执行
2. **不要急于编码**：充分的需求分析和计划制定是高质量代码的基础
3. **强制用户确认**：每个步骤都必须获得用户确认才能继续
4. **文档驱动**：每个步骤都要有明确的输出文档
5. **规范优先**：代码编写必须遵循项目编码规范
6. **验证完整性**：代码完成后必须进行审查和测试
7. **进度透明**：及时向用户反馈进度和成果
8. **选项清晰**：提供明确的选项供用户选择
